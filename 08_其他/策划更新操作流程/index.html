    策划更新操作流程（零代码版）

    给谁看：不写代码的策划同学，只想在网页上改文档、让对外分享的链接自动更新。
    你需要：能打开浏览器、有本仓库的访问权限（负责人已把你加为协作者或给了仓库链接）。
    更新日期：2026-02-08

    流程总览

    你只需要做三件事：在网页上改文档 → 提一个「合并请求」→ 等别人（或自己）点「合并」。合并完成后，网页和微信分享链接会在几分钟内自动变成最新内容，无需再操作。

    步骤 你在哪里做 做什么
    1 浏览器打开 GitHub 仓库 找到要改的那篇文档
    2 同一页面 点「编辑」→ 改字 → 提交到新分支并创建 PR
    3 PR 页面 / 请负责人 合并 PR 到 main
    4 不用管 等 2～5 分钟，网页自动更新

    下面按步骤说明（全部在网页完成，不用装任何软件、不用敲命令）。

    微信分享链接是什么？如何首次发布 / 如何更新？

    「微信链接」 = 文档站的网址。大家发在微信里的就是这一个地址，地址不变，点开看到的会随着你每次「更新发布」而变成最新内容。

    第一次还没有链接、或微信里打开是 404 / 空白

    说明文档站还没发布过，或 GitHub Pages 还没开。需要做一次性设置（通常由仓库负责人做）：

    1. 开启 GitHub Pages
      打开仓库 → 上方 Settings → 左侧 Pages → Source 选 「GitHub Actions」（本项目用 Actions 自动部署到 gh-pages，不要选 "Deploy from a branch" 里的 main）。
    2. 触发第一次部署
      仓库顶部 Actions → 左侧选 「Deploy MkDocs to GitHub Pages」 → 右侧 「Run workflow」 → 再点 「Run workflow」，等约 2～5 分钟跑完。
    3. 拿到文档站地址（即微信链接）
    4. 格式一般为：https://你的用户名.github.io/仓库名/
    5. 或在 Settings → Pages 里会显示 Your site is live at …，那个就是。
      把这个链接发到微信/群里，大家点开就是文档站；以后都用这一个链接，不用换。

    已有链接，只想「更新发布」让微信里看到最新内容

    • 方式一（推荐）：按下面步骤改文档 → 提 PR → 合并到 main。合并后约 2～5 分钟，同一链接会自动变成最新内容，微信里再点同一链接即可看到更新。
    • 方式二：不改文档，只想立刻重新发布一次（例如怀疑缓存问题）：
      仓库 Actions → Deploy MkDocs to GitHub Pages → Run workflow → Run workflow，等跑完后再用同一链接打开/分享。

    总结：微信链接 = 文档站地址，只需发布一次、发一次链接；之后每次「合并 PR」或「手动 Run workflow」都会更新该地址下的内容，不需要换新链接。

    步骤一：打开仓库，找到要改的文档

    1. 用浏览器打开本项目的 GitHub 仓库地址（负责人会发给你，形如：https://github.com/你的用户名/仓库名）。
    2. 进入 docs/ 下面的文件夹，找到对应的 .md 文件。
    3. 例如：要改「游戏概念书」，就进 01_立项企划/，点开 游戏概念书.md。
    4. 打开后你会看到当前文档的文字内容（和你在文档站里看到的对应）。

    步骤二：编辑并提交为「合并请求」（PR）

    1. 在打开的那篇文档页面，右上角找到 铅笔图标「Edit this file」，点击进入编辑模式。
    2. 在网页中的文本框里直接改字（和改 Word / 记事本一样），改完后往下滚动到页面底部。
    3. 在底部找到 「Commit changes」 这一块：
    4. 标题：可以写一句简短说明，例如「更新游戏概念书核心玩法描述」。
    5. 选项：选择 「Create a new branch for this commit and start a pull request」（为这次修改新建一个分支并创建合并请求）。
      • 不要选「Commit directly to the main branch」——我们约定只有通过「合并请求」的修改才会触发网页自动更新。
    6. 点击绿色的 「Propose changes」（或「Commit changes」后再按提示创建 PR）。
    7. 完成后会自动跳转到 Pull Request（PR） 页面，标题和说明可以再改一下，方便负责人一眼看懂你改了什么。

    步骤三：合并 PR，触发网页更新

    • 如果你有合并权限：在 PR 页面点击 「Merge pull request」 → 再点 「Confirm merge」。合并完成后，可以关掉页面。
    • 如果你没有合并权限：把 PR 的链接发给项目负责人，请对方在 GitHub 上点「Merge pull request」合并即可。
    • 合并后，GitHub 会自动重新打包文档站并更新到线上；分享出去的网页链接（包括微信里之前分享的链接）会在 2～5 分钟内变成最新内容，无需你再做任何操作。

    步骤四：确认网页已更新（可选）

    1. 打开你们平时用的文档站地址（例如 GitHub Pages 的链接）。
    2. 找到你刚改的那一页，刷新一下，确认内容已是新版本。
    3. 若在微信里分享过该链接，在微信里点开同一链接，看到的也会是更新后的内容（若没变可过几分钟再试）。

    常见问题

    问题 说明
    不知道仓库地址 向项目负责人要「代号梦山文档的 GitHub 仓库链接」；若已发过文档站链接，仓库地址一般会在同一说明里。
    找不到要改的文件 文档站左侧导航里的标题，对应仓库里 docs/ 下的 .md 文件；例如「游戏概念书」→ 01_立项企划/游戏概念书.md。
    没有「Edit」按钮 说明当前账号没有该仓库的编辑权限，需要负责人把你加为协作者（Collaborator）。
    改错了想撤销 在合并前：到 PR 页面可以继续编辑或关闭该 PR；若已合并，可联系负责人在 GitHub 上再改一版或回退。
    合并后网页没变 一般等 2～5 分钟再刷新；若超过 10 分钟仍没变，把 PR 链接或仓库的 Actions 页面发给负责人排查。

    小结

    • 你只需：在 GitHub 网页上改文档 → 用「Create a new branch + pull request」提交 → 合并该 PR。
    • 不需要：装 Git、装编辑器、敲命令行。
    • 合并到 main 的 PR 会触发自动部署，网页和微信链接会自动更新。

    更详细的发布与协作方式（含本地预览、权限与安全）见 网页发布与协作指南。
