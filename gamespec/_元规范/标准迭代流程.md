# GameSpec 标准迭代流程

> **目的**: 规范 GameSpec 自身的升级与维护过程。
> **核心工具**: `_元规范/Spec架构师助手.md`

---\n
## 1. 角色边界：你是谁？ (Who are you?)

在调用 AI 助手前，请务必搞清楚你的目标：

| 你的目标 | 你需要找谁？ | 操作对象 |
| :--- | :--- | :--- |
| **"我要设计一个游戏"** | 👷 **业务助手** (如 核心玩法助手) | 项目文档 (`项目/我的游戏/*.md`) |
| **"我要优化设计工具"** | 🏗️ **Spec 架构师助手** | 标准库 (`bin/gamespec/*`) |

> **切记**: 架构师只修印钞机，不印钞票。

## 2. 什么时候需要迭代？

当出现以下情况时，应当启动本流程：
*   **模板过时**: 现有的模板无法满足新类型游戏（如 VR、区块链游戏）的需求。
*   **流程低效**: 发现某些手动操作太繁琐，需要工具化。
*   **标准冲突**: 实际开发中发现某些规范“反人性”或逻辑自相矛盾。
*   **能力扩展**: 需要引入新的 AI 角色（如“数值平衡师”）。

---\n
## 2. 迭代工作流 (The Loop)

我们推荐使用 **"架构师助手"** 来引导这一过程，确保修改不会破坏现有结构。

### 步骤一：唤醒 Spec 架构师
1.  复制 `_元规范/Spec架构师助手.md` 的内容。
2.  发送给 AI，并附带你的需求：“大哥，我想给 GameSpec 增加一个‘Unity资源导入规范’的模板。”

### 步骤二：方案设计
架构师助手会分析你的需求，并建议：
*   新文件应该放在 `02_项目模板/资源导入规范.md`。
*   是否需要在 `01_规范标准/` 中增加对应的命名规则。
*   是否需要编写 `04_自动化工具/check_assets.py` 脚本。

### 步骤三：实施变更
根据助手的输出：
1.  **创建/修改文件**: 将内容写入对应目录。
2.  **更新引用**: 如果修改了文件名，记得运行 `04_自动化工具/` 下的链接检查脚本（如果有）或手动修复引用。

### 步骤四：更新文档
**非常重要！** 任何修改完成后，必须更新 `00_使用指南/` 下的相关文档（如《用户操作手册》），确保新人能知道这个新功能。

---\n
## 3. 版本管理建议

建议对 `bin/gamespec` 目录进行 Git 版本控制。

*   **v1.0**: 初始版本。
*   **v1.1**: 增加特定品类模板。
*   **v1.2**: 引入自动化工具链。

每次迭代后，建议在 `README.md` 中更新版本号和更新日志。
